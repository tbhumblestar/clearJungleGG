## [DS-1] 정글 풀캠프 클리어 기록 수집

### 1-1. 개요
천상계 유저들의 순수 PvE 풀캠프 클리어 기록을 수집하여, 챔피언별 최단 클리어 시간 리더보드 데이터를 구축한다.

### 1-2. 수집 대상
* **서버:** 한국(KR)
* **티어:** 그랜드마스터 + 챌린저 (약 1,000명)
  * 마스터 티어 확장은 Rate Limit 검증 후 결정 *(→ `10_BACKLOG.md` BACKLOG-005)*
* **큐 타입:** 솔로 랭크
* **포지션:** 정글(JUNGLE)

### 1-3. 배치 주기
* **매일 1회** 실행 (Spring Scheduler)
* 신기록 발견 시 실시간 알림은 추후 구현 *(→ `10_BACKLOG.md` BACKLOG-007)*

### 1-4. 타임라인 분석 (Riot Match-V5 Timeline API)

#### Step 1 — 시작 위치 판별
* 게임 시간 **1분 시점**의 유저 좌표(position)를 확인하여 시작 캠프(레드/블루/칼날부리)를 기록한다.
* 대부분의 챔피언은 정글 캠프 스폰(0:55) 직후 처치 중이므로 1분 시점 좌표가 유효하다.

#### Step 2 — 순수성 검증 (Purity Check)
4레벨 달성 전까지 외부 변수(챔피언 간 교전, 라인 미니언 등)가 개입했는지 검증한다. **하나라도 걸리면 해당 매치를 즉시 폐기한다.** 놓치는 데이터는 있어도, 오염된 데이터가 통과해서는 안 된다.

| # | 검증 조건 | 데이터 소스 | 판별 기준 |
|---|----------|-----------|----------|
| 1 | 킬/데스/어시스트 관여 | `CHAMPION_KILL` 이벤트 | 4레벨 달성 timestamp 이전의 모든 CHAMPION_KILL에서 `killerId`, `victimId`, `assistingParticipantIds`에 본인 participantId가 포함되면 폐기 |
| 2 | 라인 미니언 경험치 흡수 | `participantFrames` 스냅샷 | 4레벨 달성 직전 프레임에서 `minionsKilled > 0`이면 폐기 |
| 3 | 챔피언 대상 데미지 발생 | `participantFrames` 스냅샷 | 4레벨 달성 직전 프레임에서 `totalDamageDoneToChampions > 0`이면 폐기 |

> **1번은 이벤트 기반**이라 타임스탬프가 정확하다. **2·3번은 1분 간격 스냅샷**이라 최대 1분의 오차가 있지만, 보수적 방향(오탐으로 폐기할지언정 오염 데이터 통과 불가)이므로 허용한다.

> **못 잡는 케이스:** 챔피언에게 일방적으로 데미지를 받기만 하고 도망친 경우 (받은 피해 중 챔피언 출처를 분리하는 필드가 API에 없음). 이 경우도 클리어 타임 손실로 자연 도태된다.

#### Step 3 — 4레벨 달성 시간 추출
* 타임라인의 `LEVEL_UP` 이벤트에서 4레벨 도달 시점의 **게임 내 실제 시간(timestamp)**을 그대로 추출한다.
* 예: 게임 시계 기준 3:05에 달성 → "3:05"로 기록 및 표시.

### 1-5. DB 저장 (Upsert)

#### 복합 키 (Composite Key)
| 키 | 설명 |
|----|------|
| `team` | 레드 / 블루 |
| `startPosition` | 시작 캠프 (레드/블루/칼날부리) |
| `championId` | 챔피언 고유 ID |
| `patchVersion` | 패치 버전 |

* 동일 복합 키 기준, 기존 기록보다 빠른 경우에만 새로운 최고 기록으로 **갱신(Upsert)**.
* 같은 매치에 챌린저 정글러가 2명 있을 경우, 복합 키가 다르므로(챔피언이 다름) **양쪽 모두 기록**한다.

### 1-6. 자연 도태 (Natural Filter-out)
아래 엣지 케이스는 별도 예외 처리 없이, "최고 기록만 남긴다"는 비즈니스 룰로 자연 필터링된다.

| 엣지 케이스 | 왜 자연 도태되는가 |
|------------|-------------------|
| 페이크 동선 (인베이드 방어 등) | 이동 시간 낭비 → 클리어 타임 지연 → 1위 불가 |
| 미니언 경험치 흡수 (Leeching) | 라인 이동 시간 낭비 → 클리어 타임 지연 → 1위 불가 |

## [DS-2] 메타데이터 관리

### 2-1. Riot Data Dragon 캐싱 전략
* **챔피언 메타데이터** (`champion.json`): 패치당 1회 조회 후 DB/메모리에 캐싱. 새 패치 감지는 `versions.json`의 첫 번째 값 변경 여부로 판단.
* **룬/소환사 주문 메타데이터**: 챔피언 메타데이터와 동일하게 패치당 1회 캐싱.
* **이미지 URL**: URL 패턴만 저장하고, 실제 이미지는 Riot CDN에서 직접 서빙. 백엔드가 프록시할 필요 없음.
* 상세 URL 패턴 및 API 정보는 `6_RIOT_API_GUIDE.md` API-A ~ API-C3 참고.
